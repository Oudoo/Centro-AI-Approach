Hybrid Compute Topology

A three-tier architecture will ensure the right model runs on the right hardware — from lightweight edge inference on employee laptops to frontier-class reasoning in the cloud.

Tier 01

Edge Nodes (Laptops)

Lightweight 2B & 4B parameter models will run directly on standard company laptops (8–12 GB RAM). These will handle offline drafting, simple routing, and local summarization.

Gemma 2B Offline-First
Tier 02

Super PC Cluster

26B/31B parameter models will be hosted on centralized prosumer hardware — RTX 4090/5090 rigs (16–32 GB VRAM) and Mac Studios with custom thermal modifications for sustained inference under heavy loads.

Gemma 4 27B RTX 5090
Tier 03

Cloud Overlay

Frontier-class 70B+ MoE models will operate on RunPod & GCP VMs (40GB+ VRAM) for deep reasoning, global search indexing, and model fine-tuning workflows.

RunPod GCP A100
Remote Management via "Porta"

Section 07

Enterprise Security & Governance

Three pillars will guarantee that AI operates within the strictest boundaries of data sovereignty, access control, and auditable compliance.

Data Sovereignty

All sensitive reasoning will occur on local hardware. No PII or proprietary data will ever be transmitted to public APIs. Local Gemma 4 will ensure 100% data residency compliance.

Zero Trust Tunnels

Secure, permanent Cloudflare tunnels will bridge external mobile devices with internal IDEs and system vitals — encrypted, zero-trust remote access without exposing public ports.

MCP & RBAC

MCP will act as a strict gatekeeper. The AI will have "No God Mode" — enforcing least-privilege, requiring explicit per-operation authorization, inheriting the exact Zoho/Odoo/Genesys RBAC of the authenticated user.

Immutable Audit Logging

Every AI action — query, retrieval, tool call, and response — is appended to a write-once audit log with: User ID · Timestamp (UTC) · Session token · Input hash · Tool invoked · Output summary · Confidence score. Logs are replicated to an air-gapped store and are non-deletable by any service account, including admin roles.

Section 09B

Request Lifecycle — 7-Step Flow

Every query that enters OpenClaw traverses a deterministic pipeline before any action is taken. Here is what happens in the ~2 seconds between "send" and response.

1
Authentication & Session Bind

User identity verified via SSO token. RBAC profile loaded from Zoho People — defines which data sources and tool calls are permitted for this session.

2
Intent Classification

Edge 2B model classifies intent into one of 12 categories (query, action, report, escalate…). Ambiguous inputs trigger clarification loop before proceeding.

3
Tier Routing Decision

Complexity score determines compute tier: simple → edge, moderate → cluster, complex / frontier → cloud. Lowest capable tier is always preferred.

4
RAG Retrieval (Scoped)

Vector search against role-scoped knowledge partitions in Zoho WorkDrive. Only documents the user's RBAC grants access to are retrieved.

5
Tool-Call Gate

If action intent detected, MCP validates the requested tool call against the permission matrix. Destructive or multi-entity actions require explicit confirmation.

6
Inference + Confidence Score

Routed model generates response. Confidence score computed. If < 0.65, response is flagged with a disclaimer and placed in supervisor review queue.

7
Response Delivery + Audit Write

Response streamed to UI. Simultaneously, full event record (user, input hash, tools called, output, confidence, latency) written to immutable audit log.

Pipeline Guarantees
No Unscoped Data Access

RAG retrieval is hard-partitioned by role. A Finance user cannot surface Service Delivery agent performance records, even with a crafted query.

No Silent Actions

Every tool-call (CRM update, schedule change, email send) produces a visible confirmation step and an audit entry. Nothing happens "behind the scenes."

No Single Model Dependency

Each step can be served by a different model or tier. A cloud outage does not break steps 1–4 — only reduces the complexity ceiling.

Full Replay Capability

Because every step is logged with input hashes, any request can be fully replayed for forensic investigation — critical for compliance audits.
